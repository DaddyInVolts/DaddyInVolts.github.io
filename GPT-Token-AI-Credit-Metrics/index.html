Section 1

Problem & Opportunity

Problem Statement

The Copilot Token Consumption Scorecard is the canonical view feature crews and leadership use to evaluate the cost impact of Copilot experiments and rollouts. Today, the scorecard is missing two critical cost dimensions: GPT token consumption (input + output tokens per LLM call, by model) and AI Credit request volume (per-user/per-tenant credit balance consumption). Without these metrics on the scorecard, feature crews must pull cost data from separate dashboards or ad-hoc queries to assess whether an experiment's quality or engagement gains are justified by its cost impact — slowing ship/iterate/stop decisions and creating inconsistent cost narratives across crews.

Opportunity

Adding GPT token and AI Credit metrics directly to the existing Copilot Token Consumption Scorecard gives feature crews and leadership a single, consistent view of cost impact alongside the experimentation metrics they already trust. This:

  • Eliminates the manual cross-referencing step between cost dashboards and experiment scorecards
  • Surfaces cost regressions earlier in the experiment lifecycle (during analysis, not post-ship)
  • Provides leadership a portfolio-level view of GPT spend and AI Credit consumption per shipped feature
  • Establishes a shared cost vocabulary across feature crews, reducing inconsistent reporting

Evidence

  • Existing scorecard usage: Copilot Token Consumption Scorecard is consumed by feature crews and leadership today, indicating demand for an authoritative cost view
  • Coverage gap: Current scorecard does not include GPT token metrics or AI Credit request metrics — the two cost dimensions most commonly asked about in ship reviews
  • Decision-quality blocker: Without these metrics on the scorecard, cost impact is typically discussed verbally or via attached one-off charts during reviews
Section 2

Goals & Success Metrics

Goals

  1. Add GPT token consumption metrics (input tokens, output tokens, total tokens per LLM call, by model) to the Copilot Token Consumption Scorecard
  2. Add AI Credit request metrics (credits consumed per user, per tenant, and total per experiment cell) to the Copilot Token Consumption Scorecard
  3. Ensure both new metric families render with the same statistical rigor (CI, MDE, sample size) as existing scorecard metrics so feature crews can interpret them consistently

Section 3

User Scenarios

Scenario 1 — "Feature Crew PM Reviewing an Experiment" (Primary)

Priya is a PM on a Copilot feature crew that just completed a 2-week experiment introducing a new prompt-rewriting step before each LLM call. She opens the Copilot Token Consumption Scorecard for the experiment. Today she would see engagement and quality metrics but have to switch to a separate cost dashboard to check whether the new step inflated GPT input tokens or AI Credit usage. With this spec shipped, the scorecard now shows GPT input/output tokens (broken out by model) and AI Credit requests per user — all on the same page, with the same stat-sig framing as her other metrics. She can immediately see that input tokens grew +12% (significant) while engagement grew +3% (not significant), giving her the data to decide not to ship.

Scenario 2 — "Leadership Portfolio Review"

A Copilot org leader is preparing for a quarterly business review and wants to understand the cost trajectory across all Copilot experiments shipped in the past quarter. With these metrics on the scorecard, the leader (or their chief of staff) can pull token and credit consumption deltas for every shipped feature directly from the standard scorecard view, rather than commissioning a custom analysis. The leader can identify which features drove disproportionate token or credit growth and prioritize cost-reduction work for the next quarter.

Scenario 3 — "Data Scientist Validating Metric Definitions"

Dana is a Data Scientist validating the new metrics before GA. She confirms that GPT token counts on the scorecard match the source telemetry (within ±1% reconciliation tolerance) and that AI Credit aggregations match the credit ledger. She validates statistical computation (CI, MDE) is consistent with other scorecard metrics. She signs off and the metrics ship to all feature crews.

Edge Case — Experiments with No LLM Calls

For experiments on Copilot surfaces that don't make LLM calls (e.g., UI-only changes), the GPT token and AI Credit rows render with zero values and a small inline note: "No LLM activity observed in either variant." This avoids implying a cost issue when there is none and avoids hiding the metric entirely (which would look like a data gap).

Section 5

Design Direction — minimal additions

Placement & Layout

  • Both metric families appear as new rows in a "Cost" section of the existing Copilot Token Consumption Scorecard, grouped together for easy scanning
  • GPT token rows nest by model (one row per model in use, plus a "Total" row)
  • AI Credit rows show: Credits per User (avg), Credits per Tenant (avg), Total Credits
  • Statistical columns (CI, MDE, p-value, stat-sig flag) match the formatting used by existing rows

Tooltips & Inline Guidance

  • Each metric row has a hover tooltip with a 1–2 sentence definition and a "Learn more" link to the metric glossary
  • Tooltip explicitly notes that token/credit metrics are observational cost signals — not guardrails — to set expectations for interpretation
Section 6

Open Questions & Risks

Open Questions

Q1: What is the authoritative source-of-truth telemetry for GPT token counts?

Several pipelines emit token-related events (gateway logs, model-side telemetry, billing). Need to confirm which one the scorecard should join against to ensure reconciliation. Owner: Data Science + Engineering.

Q2: Is the AI Credit ledger queryable at the experiment-variant grain, or only at user/tenant grain?

If the ledger only aggregates by user, the scorecard will need to join credits to experiment assignment at the user level. Need to confirm with the billing/credit system team. Owner: Engineering.

Q3: How should we handle multi-model experiments where each variant uses a different model?

If treatment uses model A and control uses model B, the per-model breakdown will show different model rows per variant. Need a clear visual treatment so this isn't misread as a data gap. Owner: Design.

Q4: Are there any privacy or data-handling considerations for surfacing AI Credit data at the experiment level?

AI Credit consumption can be a proxy for paid-tier usage and may have additional handling requirements. Owner: Privacy.

Risks

Risk 1: Reconciliation Drift Between Scorecard and Source Telemetry

Likelihood: Medium · Impact: High. If scorecard token/credit values don't match the source-of-truth pipelines, feature crews will lose trust in the metrics and revert to ad-hoc dashboards. Mitigation: (1) DS validation on ≥3 sample scorecards before GA; (2) Document the reconciliation tolerance (±1%) publicly; (3) Add a scheduled reconciliation check post-GA.

Risk 2: Scorecard Render Latency Regression

Likelihood: Low–Medium · Impact: Medium. Adding new metric rows with statistical computation could slow scorecard rendering, especially on large experiments. Mitigation: (1) Pre-aggregate token and credit metrics in the analytics pipeline; (2) Run perf testing before GA; (3) Guardrail check on P95 latency.

Risk 3: Misinterpretation of Cost Metrics as Guardrails

Likelihood: Medium · Impact: Low–Medium. Feature crews may treat a "significant" token increase as a blocker even when it's not a defined guardrail, leading to inconsistent ship decisions. Mitigation: (1) Tooltip language explicitly framing these as observational cost signals; (2) Glossary entry clarifying that guardrails are a separate workstream; (3) Communicate intended use in the launch announcement.
