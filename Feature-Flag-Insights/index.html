Section 1

Problem & Opportunity

Problem Statement

When a PM, DS, or Feature Crew lead opens a scorecard in Expedite today, they see flag identifiers like chat.enableNewRanker or experiment IDs like #44231 with no human-readable description of what the flag actually changes in code. To understand what was the intervention, they have to ping the owning Feature Crew, dig through merge commits, or accept "I don't know." Cross-team scorecard review is harder than it should be, portfolio-level pattern-finding ("what kinds of changes are landing wins?") is largely manual, and the experimentation platform has no semantic understanding of the experiments running on it. The owner-written description fields in ExP/Iris and CCS exist, but they are often empty, stale, or a one-word placeholder, and there is no platform mechanism to keep them fresh as code evolves.

Opportunity

The M365 Copilot codebase is fully indexed in code search, and ≥92% of feature-flag check-sites use one of three well-known SDK shapes we can detect mechanically — isEnabled / getValue for ExP/Iris, getConfig for CCS, and a small set of in-repo gating-constant patterns. An LLM reading the code around each check-site (the call, the enclosing function, the modified branches, and the diff that introduced the flag) can emit a structured, human-readable description: what the flag gates, what changes when it's on vs. off, which code paths it affects, and which surfaces it touches. Persisted to Expedite alongside the existing experiment metadata, these inferred descriptions turn every scorecard into a self-documenting artifact, unlock portfolio queries by intervention type, and provide the semantic input that downstream specs (see Next-Iteration Recommender) need to reason over experiment outcomes.

Evidence

  • Owner-written description coverage: A sampled audit of active ExP/Iris flags in the M365 Copilot codebase found ~64% have either empty or one-line description fields ("test flag," "ranker change," "TBD").
  • Cross-team review friction: In DS partner interviews (n=14), "I don't know what this flag actually changes" was the most-cited friction point when reviewing scorecards from Feature Crews they do not actively support.
  • Scorecard-level legibility: Of 100 randomly sampled recent scorecards, 38 could not be interpreted by an outside reviewer from the flag name alone without further investigation.
  • Mechanical detectability: 100% of feature crew code lives in code-search-indexed repos already (ADO + GitHub); ≥92% of check-sites use a known SDK API we can detect with deterministic AST or regex patterns.
  • Downstream demand: The paired Next-Iteration Recommender spec explicitly depends on a machine-readable flag description being available; that work is blocked without this one.
Section 2

Goals & Success Metrics

Goals

  1. ≥90% of active ExP/Iris + CCS flags and detected in-repo gates in the M365 Copilot portfolio have an inferred description in Expedite within 24 hours of code merge or flag creation.
  2. Reviewer-rated usefulness of inferred descriptions ≥4.0/5.0 on a quarterly sample of 200 scorecards across at least 8 Feature Crews.
  3. Surface the inferred description on every scorecard row in Expedite — zero extra clicks to see "what is this flag changing."

Success Metrics

Primary Metric
Inferred-Description Coverage
Target: ≥90% of active flags · Baseline: ~36% (owner-written, often stale)
Secondary Metric
Reviewer Usefulness Score
Target: ≥4.0/5.0 on quarterly 200-scorecard sample · Baseline: not measured today
Guardrail Metric
Hallucination Rate
Must stay <3% of inferred descriptions marked "wrong" by code owner · Pause refresh if exceeded
Guardrail Metric
Freshness Lag
Median description age ≤ time-since-last-code-touch of the flag · No description older than its source code

Explicit Non-Goals

Generating the recommendation of what to do next given the description + results — that is owned by the paired Next-Iteration Recommender spec
Writing back to the ExP/Iris or CCS source-of-truth description fields — we read code and write only to Expedite's join table; owner-authored fields in source systems are untouched
Scanning third-party / out-of-portfolio repos — V1 is M365 Copilot indexed repos only
Flag lifecycle management — creation, expiry, and cleanup remain owned by the existing flag-hygiene tooling; this spec only reads
Replacing the owner-written description when one exists — we render both, owner wins when present, inferred is shown as a complement
Section 3

User Scenarios

Scenario 1 — "Cross-Team Scorecard Review" (Primary)

Priya is a PM in the Copilot Growth org. She opens an Expedite scorecard for an experiment she doesn't own — chat.enableNewRanker. Today the flag has the owner-written description "ranker change," which tells her nothing. With this spec, the scorecard row shows: "Routes Chat queries through a new GPT-4o-based ranker for the top-3 candidate completions; falls back to the existing ranker when ranker confidence is below 0.7. Affects ChatService.RankCandidates and the Excel + Word host paths." A small high-confidence pill sits next to it, and a "Why?" tooltip shows three code excerpts that informed the inference. Priya understands the intervention immediately, without pinging anyone.

Scenario 2 — "DS Portfolio Pattern Hunt"

Marcus is a DS lead doing a quarterly portfolio review. He wants to know "which intervention types are most often regressing latency guardrails?" Today this is a multi-day exercise of pulling 80 scorecards and manually categorizing them. With inferred descriptions and the tag taxonomy (R-08), Marcus filters Expedite to "intervention tag = ranking change" and instantly sees the 23 ranker experiments shipped in the last quarter, with their regression patterns side by side. He builds his portfolio brief in an afternoon instead of a week.

Scenario 3 — "Flag-Hygiene Cleanup"

Maya owns a Feature Crew with ~140 active flags. She opens the FC-level coverage dashboard (R-07) and sees that 12 of her flags have an inferred description reading "No active references found in indexed repos as of [date]." Each one is a hygiene candidate — the flag still exists in ExP/Iris but the code that checked it has been removed. She opens a cleanup work item for each, retires the flags, and reduces her crew's flag-management burden.

Scenario 4 — "Owner Correction Loop"

Jordan reviews the inferred description for one of his own flags and disagrees with it — the inference mis-attributed the affected surface. He clicks "Edit" on the row, types the correct description in 30 seconds, and saves. The override (a) wins for all future scorecard displays of that flag, (b) is captured as a hallucination-feedback signal for model evaluation, and (c) is preserved across re-scans until Jordan removes it. His correction does not write back into ExP/Iris source-of-truth.

Edge Case — "Generated code only"

A flag has check-sites that exist only inside generated code (e.g., proto-generated stubs, codegen scaffolds). The scanner detects the generated-file markers, excludes those check-sites from inference, and emits a "No human-authored references found" description with a low-confidence pill. The flag still appears in Expedite — but the description tells reviewers honestly that we couldn't infer meaning from the code we found.

Section 4

Requirements

Priority ID Requirement Acceptance Criteria Notes
P0 R-01 Scan all indexed M365 Copilot repos (ADO + GitHub) for feature-flag check-sites across three source classes: ExP/Iris (isEnabled, getValue, getTreatmentVariable), CCS (getConfig and equivalent SDK shapes), and in-repo gating constants (boolean / enum constants whose name matches the configured gate-naming convention and that gate a code branch). Emit a structured record per check-site with file, function, flag ID, source class, and the surrounding 40-line context window. ≥92% of historically known ExP/Iris and CCS check-sites recovered on a labeled validation set of 500 sites; in-repo gate recall ≥75% on labeled set; zero false positives in generated-code paths. Reuses existing code-search index
P0 R-02 For each detected flag, generate a single inferred description using an LLM over (flag ID + every detected check-site's 40-line context + the diff at flag introduction when available + the enclosing function/method signature). Description schema: "what the flag gates" (1 sentence), "on vs. off behavior" (1 sentence), "affected code paths" (file+function list, max 5), "surfaces touched" (host list — Chat / Excel / Word / PowerPoint / Outlook / etc.). ≥90% of generations conform to the schema; ≥80% of generations on a 200-flag eval set are rated ≥4/5 by the flag's owning engineer; refusal-rate on ambiguous input <5%. Model + prompt versioned and pinned
P0 R-03 Persist inferred descriptions in Expedite's experiment-metadata join table, keyed on (flag_id, scan_run_id). Refresh on relevant code change: a repo webhook triggers re-scan of any flag whose check-sites touch the changed files; refreshed description appears within 24 hours of code merge. End-to-end webhook → updated row in Expedite ≤24h on ≥99% of triggering merges; freshness telemetry visible in IDEAS dashboards. Idempotent on flag_id
P0 R-04 Surface the inferred description on every scorecard row in Expedite that has a matched flag: a new "What this flag changes" field directly under the flag ID, with a confidence pill (high / medium / low), the affected-surface chips, and a "Why?" tooltip exposing the top-3 code excerpts that informed the inference plus the model + prompt version. Field renders on 100% of scorecard rows with a matched flag; confidence pill present; "Why?" tooltip opens with three excerpts plus model/prompt version; Fluent UI accessibility (keyboard nav, screen reader) verified. Coordinate with Expedite frontend
P0 R-05 Every inferred description renders with a visible "AI-generated — verify before quoting" disclaimer in the scorecard UI. Owners can edit/override the inferred description per flag via an in-line edit affordance; the override wins over inference for all future displays and survives re-scans until the owner removes it. Disclaimer visible on 100% of inferred rows; edit-and-save round-trips and persists across re-scans; override events emit a feedback signal; edit history available in an audit panel.
P1 R-06 Hallucination-feedback path — any reviewer can mark an inferred description "Wrong" or "Outdated" via a one-click affordance on the scorecard row. Feedback writes a structured record (flag_id, reviewer, reason, optional free-text) into a model-evaluation table that feeds the quarterly prompt + eval-set refresh. Feedback persists; weekly aggregate visible to inference-model owners; quarterly eval pipeline ingests the feedback table; reviewer can see "your feedback was incorporated in eval vN+1" on the next refresh.
P1 R-07 Coverage dashboard for FC leads — per-Feature-Crew view of (a) % of FC flags with inferred descriptions, (b) % overridden by owners, (c) % flagged as wrong, (d) % with "no active references found." Linked from the FC's main Expedite landing page. Dashboard refreshes daily; all four counters present; deep-links from each counter to the underlying flag list with filters pre-applied.
P1 R-08 Tag the inferred description with a small fixed taxonomy of intervention categories — ranking, prompt, latency, ui, auth, telemetry, rollout-mechanic, model-routing, caching, throttling, content-filter, host-targeting, other. Tags are inferred by the same model in the same call and used by downstream consumers (portfolio queries, the Next-Iteration Recommender). ≥85% of inferred descriptions have at least one tag; tag-precision on labeled 200-flag eval set ≥0.80 (macro F1); taxonomy frozen for V1 (no per-FC custom tags). Taxonomy reviewed by IDEAS DS
P2 R-09 Surface the inferred description in the ExP/Iris flag-detail UI as a read-only "Expedite says" panel — clearly labeled, one-way (we do not write back to ExP/Iris source-of-truth), with a deep link back to the Expedite row. Panel renders on the ExP/Iris flag-detail page when an inferred description exists; "Edit in Expedite" link round-trips; no write paths to ExP/Iris from this surface. Coordinate with ExP/Iris team
P2 R-10 Inference observability — emit metrics for scan latency, coverage by source class (ExP/Iris vs. CCS vs. in-repo gate), hallucination rate from R-06 feedback, override rate from R-05, and refresh lag from R-03 webhooks. All five metrics visible in IDEAS telemetry within 60 days of GA; alert on coverage drop >2% week-over-week or hallucination rate exceeding 3%.
Section 5

Design Direction

End-to-End Flow

Code Merge / New Flag Created  →  Repo Webhook Fires  →  Re-Scan Scheduled for Affected Flags
→  Code Scanner Walks Indexed Repos  →  Per-Flag Check-Site Bundle Built
→  LLM Generates Structured Description + Intervention Tags  →  Confidence Scored
→  Persisted to Expedite Join Table  →  Scorecard UI Renders Description + Confidence Pill + "Why?" Tooltip

Owner can edit/override at any time  →  Override wins on subsequent displays  →  Hallucination feedback feeds quarterly eval refresh

Confidence Pill Rules

High ≥3 active check-sites, low variance across them, code touched within last 60 days, flag introduction diff available. The model is reading consistent, recent, authoritative code.
Medium 1–2 check-sites OR moderate variance across sites OR last code touch 60–180 days ago. Description is likely correct but worth a glance before quoting in a shipping doc.
Low Flag name + owner-written description only (no check-sites found), OR check-sites in generated code only, OR last code touch >180 days ago. Treat as a hint, not a fact.

Scorecard Row Treatment

  • Inline description: The "What this flag changes" line appears immediately under the flag ID on every scorecard row, with the confidence pill, intervention tag chips, and host chips on the same line.
  • "Why?" tooltip: Hover or focus the confidence pill to see the top-3 code excerpts that drove the inference, the model + prompt version, and a deep link to each excerpt in the code-search index.
  • Owner-written + inferred coexistence: When ExP/Iris or CCS has a non-empty owner-written description, both are shown — owner-written first, labeled "Owner says"; inferred labeled "Inferred from code". If they materially disagree, a small "disagrees with owner description" warning chip appears.
  • Edit affordance: Pencil icon at the end of the row opens an in-line editor. Save persists the override; clear-override re-enables inference. Edits never write back to ExP/Iris or CCS source-of-truth.

Edge Cases & Empty States

  • No check-sites found in code: Description reads "No active references found in indexed repos as of [date]." with a low-confidence pill and a "Flag hygiene candidate" chip. Surfaces in the FC dashboard's hygiene counter (R-07).
  • Generated-code-only references: Excluded from inference. Description reads "References exist only in generated code — inference skipped." with low confidence.
  • Multi-host flag with divergent check-sites: Per-host description fragments are joined into one inferred description and the host chips reflect every detected host; if check-sites disagree on behavior per host, the disagreement is called out explicitly in the description body.
  • Owner override matches inference: No visible difference to reviewers; override is persisted but the system tracks it as "owner-confirmed inference" for eval-quality tracking.
  • Owner override conflicts with inference: Override wins on display. Conflict is captured for model evaluation (high-signal correction). After 90 days, if the underlying code has not changed and the override has not been re-confirmed by the owner, a "Re-confirm or release override" prompt is shown to the owner in the FC dashboard.
  • Sensitive repos (privacy-flagged areas): Scanner respects repo ACLs; inference is skipped for code the inference service is not authorized to read; row shows "Description unavailable in this scope" with no leakage of code content.
  • Model service outage: Scanning still runs and persists check-site records; descriptions are generated when the inference service recovers. Scorecard row shows "Description pending" rather than a stale or empty placeholder.
Section 6

Open Questions & Risks

Open Questions

Q1: Which inference model — internal Copilot model, AOAI deployment, or IDEAS-hosted?

Each has trade-offs across cost, latency, data boundary, and quality on code. The candidates need a head-to-head bake-off on a 500-flag labeled eval set before commit. Owner: David Lydston with IDEAS Eng + Copilot ML. Current leaning: Bake-off in dogfood; default to the option with the lowest hallucination rate on the eval set even if not the cheapest.

Q2: Do we infer for flags that already have a non-empty owner-written description?

Options: (a) infer always, show both, owner wins on conflict; (b) infer only for empty descriptions; (c) infer always but suppress inference display if owner description is >X chars. Owner: David Lydston with a sample of FC leads. Current leaning: (a) — always infer, always show both, owner wins; the cost of always inferring is low and the value of disagreement detection (a quality signal) is high.

Q3: How do we treat in-repo "feature constant" gates that aren't tied to any ExP/Iris ID?

These exist (boolean const + if (const)) but have no rollout machinery. Do we surface them as their own pseudo-flag rows in Expedite? Owner: David Lydston with Expedite PM. Current leaning: Yes — emit them with a clear "in-repo constant — not a managed flight" badge so reviewers can see what is gating behavior without confusing them with experimentable flags.

Q4: Refresh budget — re-scan every flag on every triggering merge, or batch nightly?

Webhook-driven keeps freshness tight but costs more inference calls; batching nightly is cheaper but breaks the 24h SLA on fast-iteration days. Owner: David Lydston with IDEAS Eng. Current leaning: Hybrid — webhook-driven for any flag currently associated with an active rollout in Expedite; nightly batch for the long tail.

Q5: Cross-FC visibility of inferred descriptions — universal-read or per-FC ACL?

Today, scorecard data in Expedite is generally cross-FC visible. Inferred descriptions may surface implementation details (function names, surface routing) that some FCs treat as sensitive. Owner: David Lydston with Privacy Review and a sample of FC leads. Current leaning: Universal-read for V1 (matches existing scorecard model); allow FCs to mark individual flags "description hidden cross-FC" as a P2 follow-up if needed.

Risks

Risk 1: Hallucinated descriptions quoted as ground truth

Likelihood: Medium · Impact: High. If a PM quotes an inferred description in a ship-decision doc and the description is wrong, real bad decisions follow. Mitigation: (1) Persistent "AI-generated — verify before quoting" disclaimer in the UI (R-05); (2) Confidence pill with clear low-confidence visual treatment; (3) Owner edit/override path with the override winning permanently (R-05); (4) Hallucination feedback loop into quarterly eval refresh (R-06); (5) Hard guardrail metric — pause refresh if hallucination rate exceeds 3%.

Risk 2: Code data boundary — inference service reads code it shouldn't

Likelihood: Low · Impact: High. The scanner and inference service must respect repo ACLs and stay within the M365 data boundary; a misconfiguration could send privileged code to an unauthorized endpoint. Mitigation: (1) Scanner runs as a principal with read-only access scoped to indexed-repo ACLs; (2) Inference endpoint is in-region with no cross-tenant pooling of code in a single prompt; (3) Privacy Review sign-off before GA; (4) Audit log of every (flag_id × code excerpt × model call) tuple, retained per existing compliance retention rules; (5) Annual third-party privacy review.

Risk 3: Index drift erodes freshness during fast iteration

Likelihood: Medium · Impact: Medium. On heavy-merge days, webhook backlog could push refresh past the 24h SLA, and a stale description displayed without a "stale" indicator misleads reviewers. Mitigation: (1) Webhook-driven hybrid refresh (Q4); (2) Surface description freshness on the row with explicit "Refreshed X hours ago" text; (3) "Pending refresh" badge when a triggering merge is detected but the refresh hasn't completed; (4) Telemetry alert on refresh lag >24h for any active-rollout flag.

Risk 4: Reviewer over-trust degrades over time

Likelihood: Medium · Impact: Medium. Day 1, reviewers will be appropriately skeptical of AI-generated descriptions; over time, the disclaimer becomes invisible and inferred descriptions get treated as authoritative. Mitigation: (1) Disclaimer is persistent and visually present, not a dismissible toast; (2) Confidence pill is colorized and prominent for low-confidence rows; (3) Quarterly usefulness survey (Goal 2) explicitly asks reviewers whether they verified the description before acting on it; (4) Random 5% of high-confidence descriptions are spot-audited by IDEAS DS each quarter and any false high-confidence is treated as a hallucination-rate guardrail breach.

Risk 5: In-repo gate detection misclassifies non-flag constants

Likelihood: Medium · Impact: Low. The in-repo gating-constant detector may pull in boolean constants that aren't actually feature gates (e.g., debug toggles, internal-only switches). False positives add noise to Expedite. Mitigation: (1) Conservative initial naming convention — require the constant to gate a top-level if branch with both arms non-empty; (2) Per-FC ability to suppress specific constants from inference; (3) Pseudo-flag rows carry the "in-repo constant — not a managed flight" badge so reviewers know not to treat them as experimentable; (4) FC dashboard surfaces detected-gate count so FCs can spot noise and report it via R-06 feedback.
