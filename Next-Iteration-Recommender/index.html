Next-Iteration Recommender for Expedite

Mini Spec — Given the inferred flag description and the experiment results, propose what the next iteration should change
PM Owner David Lydston
Team IDEAS Experimentation
Last Updated June 23, 2026
Status Draft
Stakeholders Expedite, IDEAS DS, M365 Copilot Feature Crews, Copilot ML, Privacy Review, Growth Orchestrator
Depends on
  • Feature Flag Insights in Expedite — provides the machine-readable inferred description of what each flag changes and the intervention-tag taxonomy. Without this, the recommender has only a flag name to reason over.
  • Guardrail Decision Framework — provides the deterministic not-safe / extend verdict per guardrail and the regression detection signal that triggers a recommendation set.
  • Growth Orchestrator Metadata in Expedite — provides the canonical hypothesis, initiative classification, and Feature Crew ownership used to compare recommendations against owner intent and route follow-ups correctly.
Section 1

Problem & Opportunity

Problem Statement

When an M365 Copilot experiment regresses a guardrail or fails to move its primary metric, the Feature Crew's next move today is "look at the scorecard, talk to DS, guess at a follow-up." There is no platform-side proposal for "given what your flag changed and how it landed, here's a starting menu of what V2 should change differently." The result varies wildly across teams: strong FCs run rigorous iteration loops, weaker ones ship V1 and quietly move on when regressions appear. Of regressing experiments, ~41% have no documented follow-up experiment within 60 days, and only ~12% of regressions get a follow-up that specifically targets the guardrail that regressed. The platform has all of the inputs needed to do better — inferred flag description, scorecard results, guardrail verdict, and a rich archive of past similar interventions — but they have never been joined into a recommendation surface.

Opportunity

With (a) Feature Flag Insights surfacing the what (inferred description + intervention tags), (b) the Expedite scorecard supplying the result, and (c) the Guardrail Decision Framework specifying the failure mode, an LLM-driven recommender can read all three together and propose a ranked list of concrete next-iteration changes. Each recommendation is tied to a written hypothesis, a small set of similar past experiments retrieved from the Copilot portfolio archive, an expected impact direction, and a suggested primary guardrail to watch on the follow-up flight. Owners get a starting menu they can accept (registers a follow-up flight shell pre-populated with metadata), modify, or reject with reason — and the rejection signal feeds back into model evaluation. The portfolio learns from acceptance over time, and FCs get a consistent, evidence-grounded starting point for iteration instead of an empty page.

Evidence

  • No-follow-up rate: ~41% of experiments that ship a regression have no documented follow-up experiment registered within 60 days (sampled portfolio audit, last four quarters).
  • Targeted-follow-up rate: Of regressions that do receive a follow-up, only ~12% have a follow-up that specifically targets the guardrail that regressed; the rest target unrelated metrics or repeat the original intervention.
  • FC retro signal: "We didn't know what to try next" is the most-cited reason for not iterating in published M365 Copilot FC retros over the last six months.
  • Archive richness: The Copilot portfolio archive contains 8,000+ past experiments with structured intervention + guardrail outcome data — sufficient retrieval depth for a recommender.
  • Time-to-follow-up: Median time from regression detection to a registered follow-up flight is ~38 days today; FC retros consistently target ≤14 days as the threshold for "iteration loop is healthy."
Section 2

Goals & Success Metrics

Goals

  1. Every experiment with a not-safe guardrail verdict gets a ranked list of ≥3 candidate next-iteration changes in the Expedite detail view within 24 hours of the cadence run that produced the verdict.
  2. Owner acceptance rate (Feature Crew marks a recommendation as "trying this," registering a follow-up flight shell) reaches ≥30% on Sev≤2 regressions within 60 days of GA.
  3. Median time from regression detection to a registered follow-up experiment drops from ~38 days to ≤14 days within 90 days of GA.

Success Metrics

Primary Metric
Recommendation Acceptance Rate
Target: ≥30% of recommended next iterations registered as a follow-up flight · Baseline: 0% (no recs today)
Secondary Metric
Time-to-Follow-Up
Target: median ≤14 days regression → registered follow-up · Baseline: ~38 days
Guardrail Metric
Worse-Than-Nothing Rate
Must stay <5% of recommendations rated by owner as "actively misleading" · Halt recs for that FC if exceeded
Guardrail Metric
Diversity Floor
Every recommendation set covers ≥2 distinct intervention categories · Hard rule, not a soft target
Outcome Metric
Recommendation-Driven Fix Rate
Target: ≥40% of accepted recommendations land a follow-up that closes the original regressed guardrail · Baseline: not measured

Explicit Non-Goals

Auto-running the recommended iteration without owner approval — every accepted recommendation creates a follow-up shell that the owner reviews, designs, and ramps via the existing flight workflow
Inferring the flag description — owned by the upstream Feature Flag Insights spec; this spec consumes the inferred description and does not generate it
Replacing the FC's design judgement — recommendations are a starting menu, not a verdict or a mandate
Recommending follow-ups for neutral or winning experiments at V1 — V1 focuses on regression follow-up; expanding to "how to amplify a win" is a V2 candidate
Cross-portfolio recommendations that leak intervention specifics across FC trust boundaries — retrieval respects per-FC opt-in for cross-FC intervention details
Section 3

User Scenarios

Scenario 1 — "Sev2 Latency Regression Follow-Up" (Primary)

Maya's Feature Crew shipped a new Chat ranker (chat.enableNewRanker) and the 9:00 AM cadence returned a not-safe verdict on the Chat p95 latency guardrail — regression of +180ms localized to Excel-host queries. Within 24 hours, a Suggested next iterations panel appears on the experiment's Expedite detail page. The recommender has read the inferred description ("Routes Chat queries through new GPT-4o ranker for top-3 candidates"), the regression detail (p95 +180ms, Excel-host only), and the archive (3 prior latency-regressing ranker changes — 2 fixed by adding a cache layer, 1 fixed by tiering by confidence). It proposes three options: (1) tier the new ranker by confidence — fall back to the old ranker below 0.7 confidence; (2) add a query-result cache layer at the top of the ranker pipeline; (3) carve out Excel host from the new ranker until a host-specific perf investigation completes. Each recommendation lists the 3 most-similar past experiments. Maya clicks "Try this" on option 1 — a follow-up flight shell appears in her FC's flight queue with the hypothesis, suggested guardrail, and analog past flight links pre-populated. She modifies the cohort plan and ramps.

Scenario 2 — "Multi-Host Single-Surface Regression"

Priya's flag regresses an engagement guardrail on Word-host queries but is neutral on Excel and PowerPoint. The recommender — using the host chips from the inferred description plus the per-host scorecard breakdown — proposes a host-specific carve-out as the top recommendation, citing 4 past flags that resolved similar single-host regressions with host-targeted rollbacks. Priya accepts; the follow-up shell is pre-populated with a Word-only audience filter.

Scenario 3 — "Owner Hypothesis Disagreement"

Jordan's experiment was registered in Growth Orchestrator with the hypothesis "this will improve response satisfaction by reducing latency." The flag shipped, latency moved as expected, but response satisfaction regressed. The recommender surfaces this disagreement explicitly at the top of the panel: "Owner hypothesis was that latency improvement would lift satisfaction; experiment improved latency but regressed satisfaction. Recommendations target satisfaction directly, not latency." The three recommendations then target the satisfaction signal rather than the original latency hypothesis.

Scenario 4 — "Repeated Regression on the Same Flag"

This is the third iteration of the same intervention area; the flag has regressed the same guardrail on V1, V2, and V3. The recommender escalates: the top-ranked option is now "Consider stopping this intervention — three consecutive iterations have failed to clear this guardrail; the archive shows similar three-strike patterns rarely recover." The other two recommendations remain change-oriented but the stop-option is the prominent top result.

Edge Case — "No analogous past experiments"

For a novel intervention category, archive retrieval cannot find ≥2 strong analog past experiments. The recommender still emits recommendations but caps the set at 2 instead of 3, applies a low-confidence label to each, and prepends [Exploratory — no strong analog] to each option's hypothesis. The owner is told honestly that they are in less-charted territory.

Edge Case — "Inferred description missing"

Feature Flag Insights does not (yet) have a description for the flag — the upstream coverage gap. The recommender uses the scorecard + guardrail verdict + flag-name embedding only, returns recommendations with a clearly visible "Low confidence — no flag description available" badge, and includes a deep link to request inference for the flag. Owners can still accept or modify, but they're told the input was incomplete.

Section 4

Requirements

Priority ID Requirement Acceptance Criteria Notes
P0 R-01 Trigger — on every cadence run that produces a not-safe verdict (per Guardrail Decision Framework), compute and persist a recommendation set within 24 hours of cadence completion. Re-compute on subsequent cadence runs if the verdict persists and the underlying inputs (description, scorecard, archive) have changed materially. ≥99% of qualifying verdicts have a persisted recommendation set within 24h in dogfood; re-compute on input change verified on synthetic flip-flop test. Depends on Guardrail Decision Framework
P0 R-02 Recommendation set — 2 to 5 ranked candidate next-iteration changes. Each candidate carries: a one-line hypothesis, a 2–3 sentence change description, top-3 most-similar past experiments (with deep links), expected impact direction on the regressed guardrail, the suggested primary guardrail to watch on the follow-up, and a confidence label (high / medium / low). All six fields present on ≥99% of recommendations; rank order stable across model temperature variation on a labeled eval set; cap at 5 enforced. Model + prompt versioned
P0 R-03 Inputs to the model — (a) inferred flag description + intervention tags from Feature Flag Insights, (b) full scorecard with all guardrail verdicts and direction-of-effect from Guardrail Decision Framework, (c) Growth Orchestrator hypothesis + initiative classification, (d) top-N retrieval from the Copilot portfolio archive on (intervention tag × regressed guardrail × surface) similarity. Retrieval depth N is a tunable hyperparameter, default 20. End-to-end pipeline produces a recommendation from these four inputs in <90 seconds on the 95th percentile; missing inputs gracefully degrade per edge-case rules. Retrieval embeddings pre-computed nightly
P0 R-04 Surface in Expedite — a "Suggested next iterations" panel on the regressing experiment's detail page, with recommendations ranked, expandable, and an explicit "AI-generated; verify before acting" disclaimer at the top of the panel. Surface a count badge in the experiment's row in the FC's flight list. Panel renders on 100% of qualifying experiments; disclaimer visible; count badge present in flight list; Fluent UI accessibility verified. Coordinate with Expedite frontend
P0 R-05 Owner actions per recommendation — "Try this" (creates a follow-up flight shell with hypothesis, suggested guardrail, parent flight link, and analog past flights pre-populated; routes to the FC's flight intake queue), "Modify" (opens an editable form with the recommendation pre-loaded), "Reject with reason" (captures structured reason from a small fixed list: not feasible, misreads what we changed, wrong analog, conflicts with strategy, other). All three actions round-trip; "Try this" produces a follow-up flight shell in <5 seconds; rejection reasons persist; FC can see their own action history.
P1 R-06 Outcome feedback loop — when a follow-up flight created via "Try this" runs to a result (cadence verdict reached), capture whether the follow-up actually closed the originally regressed guardrail. Aggregate into the Recommendation-Driven Fix Rate metric and surface a per-FC dashboard. End-to-end outcome capture works on dogfood; per-FC dashboard shows fix rate refreshed weekly; outcome signal feeds back into the model evaluation pipeline. Long-feedback-loop metric
P1 R-07 Diversity floor — recommendation engine guarantees ≥2 distinct intervention categories per recommendation set. If only one category has strong analogs, force the second slot to a different category even at lower confidence, labeled accordingly. Avoid mode collapse where every option is "add a cache." Synthetic test: 100 generated recommendation sets exhibit ≥2 distinct tags in 100% of cases; manual audit of 50 production sets shows no category-collapse complaints. Diversity check is post-generation filter
P1 R-08 Confidence labels — each recommendation is labeled high / medium / low based on (a) count of strong analog past experiments, (b) model confidence, and (c) whether inferred flag description was available. Rules pinned in the rule dictionary alongside the Decision Framework rules. Confidence rules documented and pinned; manual eval on 200 recommendations shows ≥80% agreement between human raters and the rule-based label.
P1 R-09 Surface the top recommendation in the auto-filed ADO bug (per the Auto-File Bug / IcM for FCs on Guardrail Violations spec) — a "Suggested next iteration" section in the bug body with the top option, hypothesis, and a deep link back to the full panel in Expedite. Top recommendation appears in every qualifying bug body; deep link round-trips to Expedite; only the top is included to keep bug body scannable. Coordinate with FC-Guardrail-Bug-Filing spec
P2 R-10 Per-FC opt-out — FCs can opt their experiments out of recommendations at the experiment-creation or flight-creation stage (e.g., highly sensitive experiments where an AI-suggested next move is inappropriate). Opt-out is logged and visible to the FC lead in the per-FC dashboard. Opt-out flag persists; recommender skips opted-out experiments; FC dashboard counter shows opted-out count.
P2 R-11 Recommender observability — emit metrics for acceptance rate, time-to-follow-up, worse-than-nothing rate, diversity coverage, generation latency, and recommendation-driven fix rate (R-06) to IDEAS telemetry. All six metrics visible in IDEAS telemetry within 60 days of GA; alert wired on worse-than-nothing rate >5% per FC.
Section 5

Design Direction

End-to-End Flow

Not-Safe Verdict from Guardrail Decision Framework  →  Recommender Worker Triggered
→  Fetch Inputs (Inferred Description + Scorecard + Orchestrator Hypothesis + Archive Retrieval)
→  LLM Generates Ranked Recommendations  →  Diversity Filter Applied  →  Confidence Labels Assigned
→  Persisted to Expedite  →  Surfaced in Detail Page Panel + Auto-Filed Bug Body + FC Flight-List Badge

Owner chooses "Try this" / "Modify" / "Reject with reason"  →  Follow-up flight shell created  →  Outcome at next cadence feeds fix-rate metric & eval refresh

Ranking Rules

  • Primary signal: Number and strength of analog past experiments that closed a similar regression with the candidate intervention type.
  • Secondary signal: Model confidence on the (description, regression, intervention) tuple.
  • Tertiary signal: Estimated effort to implement (proxied by intervention category — e.g., a host carve-out is cheap; a cache layer is moderate; an architecture change is heavy). Lower effort breaks ties when other signals are equal.
  • Diversity post-filter (R-07): After ranking, the top-ranked option in each distinct intervention category gets a guaranteed slot; remaining slots fill by rank.
  • Repeated-regression escalation: If this is the third iteration of the same intervention area with the same regression, the "Consider stopping this intervention" option is force-promoted to rank 1.

Edge Cases & Empty States

  • No inferred flag description (upstream gap): Recommend using scorecard + guardrail + flag-name embedding only. Surface with a "Low confidence — no flag description available" badge and a deep link to request inference. Cap set size at 2.
  • No strong historical analogs: Return up to 2 recommendations with low-confidence labels and an [Exploratory — no strong analog] prefix on each hypothesis. Honest signaling beats fabricated specificity.
  • Owner hypothesis disagreement: If the experiment regressed a metric the owner's hypothesis did not target, surface the disagreement explicitly at the top of the panel and target recommendations at the regressed metric, not the owner's original hypothesis target.
  • Repeated regression on the same intervention area (3+ iterations): Top-ranked recommendation is "Consider stopping this intervention" with the analog evidence cited. Other change-oriented recommendations remain in the set but below.
  • Cross-FC archive retrieval restricted by opt-out: Recommender retrieves only the public summary (intervention tag + outcome direction) for restricted past flights; intervention specifics are redacted. The recommendation card shows "Analog details redacted (cross-FC privacy)" instead of full hypothesis text.
  • FC opted out (R-10): No recommendations generated or surfaced. The opt-out is visible to the FC lead on the per-FC dashboard for audit.
  • Model service outage: Recommendation panel shows "Recommendations pending — retrying" with the last-known set (if any) visible read-only. Outage exceeding 4 hours emits an IcM to the recommender on-call.
  • Worse-than-nothing rate exceeds guardrail per FC: Recommendation surfacing is paused for that FC pending model + retrieval review. The FC is notified explicitly so the absence is not silent.
Section 6

Open Questions & Risks

Open Questions

Q1: Opt-in default vs. opt-out default per FC

Should recommendations surface for every regressing experiment by default (opt-out, R-10), or require an FC lead to explicitly opt their crew in? Opt-in is safer but slows adoption; opt-out gets coverage fast but risks FCs feeling AI-suggested next moves are imposed. Owner: David Lydston with Copilot Eng leadership. Current leaning: Opt-out (surfaced by default) for V1, with a single-click per-experiment opt-out at flight creation. Re-evaluate at the 90-day post-GA mark based on worse-than-nothing rate.

Q2: Cross-FC archive retrieval policy

The strongest analogs for a given regression often live in other FCs. Some FCs treat intervention specifics as private. How much is retrievable cross-FC, and what is redacted? Owner: David Lydston with Privacy Review and FC leads. Current leaning: Cross-FC retrieval enabled by default for metadata (intervention tag, regression type, outcome direction); intervention-specific details require source-FC opt-in; redacted analogs still count toward analog support but show "details redacted" in the recommendation card.

Q3: Where does the model actually run?

Internal Copilot model, AOAI deployment, or IDEAS-hosted? Cost + latency + data-boundary trade-offs differ. Recommender prompts are larger than Feature Flag Insights prompts (scorecard + archive context) which amplifies cost differences. Owner: David Lydston with IDEAS Eng + Copilot ML. Current leaning: Same provider chosen for Feature Flag Insights (Q1 there) to share the data-boundary review; benchmark cost separately because of larger prompt size.

Q4: Recommend on extend verdicts as well as not-safe at V1?

The trigger (R-01) covers not-safe only. Extend verdicts also represent regressions in spirit (CI overlap with the practical-significance threshold); recommending on extend would expand coverage but increase noise. Owner: David Lydston with Guardrail Decision Framework owner. Current leaning: Not-safe only at V1; revisit for V1.5 once acceptance rate stabilizes.

Q5: Does the model see PII or sensitive metric values?

The scorecard input includes guardrail values (e.g., Harmful Content Rate) and the archive includes the same. Some guardrail metrics carry sensitive aggregate signals. Do we redact, summarize, or send raw? Owner: Privacy Review + IDEAS DS. Current leaning: Summarize guardrail outcomes as direction + magnitude bucket (e.g., "regression: 100–250ms p95") rather than raw values for the recommender's prompt; raw values remain available only for the scorecard surface itself.

Risks

Risk 1: Mode collapse — every recommendation is "add a cache"

Likelihood: High without explicit mitigation · Impact: High. If archive retrieval is weak or the model defaults to a small set of high-prior interventions, recommendations become boilerplate and FCs stop reading them. Mitigation: (1) Hard diversity floor — ≥2 distinct intervention categories per set (R-07); (2) Per-category retrieval depth tuned to surface uncommon successful interventions; (3) Monthly red-team review of 50 recent recommendation sets by IDEAS DS; (4) Repeated-recommendation detection — if the same hypothesis pattern is recommended on >20% of regressions in a 30-day window, alert the model owners.

Risk 2: Owner over-trust — FCs ship recommendations without independent design review

Likelihood: Medium · Impact: High. AI-suggested next moves accepted without scrutiny could produce real downstream regressions. Mitigation: (1) Persistent "AI-generated; verify before acting" disclaimer on every panel (R-04); (2) "Try this" creates a flight shell, not a ramped flight — the owner still designs and approves the rollout; (3) Confidence labels with clear low-confidence visual treatment (R-08); (4) Outcome feedback loop (R-06) tracks whether accepted recommendations actually fixed the regression — published per FC so under-performing acceptance patterns are visible.

Risk 3: Regressive feedback loop on acceptance signal

Likelihood: Medium · Impact: Medium. If acceptance is treated as the headline quality metric, the system optimizes for "things owners click yes on" rather than "things that actually fix the regression." Mitigation: Ground the quality metric in outcome, not click-acceptance — Recommendation-Driven Fix Rate (R-06) is the metric the recommender is optimized against in eval; acceptance rate is reported but not the optimization target.

Risk 4: Cross-FC retrieval leaks intervention secrets

Likelihood: Medium without policy · Impact: High. The recommender's analog-experiments list could surface intervention details that other FCs treat as confidential, violating internal trust boundaries. Mitigation: (1) Per-FC opt-in for cross-FC retrieval of intervention specifics (Q2); (2) Default redaction of intervention specifics for non-opted-in FCs — redacted analogs show only tag + outcome direction; (3) Audit log of every cross-FC retrieval, available to FC privacy leads; (4) Quarterly privacy review of retrieval logs.

Risk 5: Latency budget for 24h SLA blown by large archive retrieval

Likelihood: Medium · Impact: Medium. Real-time embedding retrieval over an 8,000+ flight archive plus LLM generation may not consistently fit a 24h SLA at scale, especially as the archive grows. Mitigation: (1) Pre-compute retrieval embeddings nightly (R-03); (2) Batch generation in scheduled windows aligned to cadence-run completion peaks; (3) "Recommendations pending" placeholder rather than a degraded set if generation fails; (4) Capacity model reviewed quarterly against archive growth.

Risk 6: Recommendations contradict an in-flight override from the Decision Framework

Likelihood: Low · Impact: Medium. If an FC has filed a documented override accepting a guardrail regression for known reasons, surfacing recommendations to "fix" the regression would undermine the override flow. Mitigation: Recommender reads the override state from the Decision Framework; experiments with an accepted override get a banner "This experiment has an accepted override — recommendations are informational only" and the panel collapses by default.
